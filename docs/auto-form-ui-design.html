Vision createAutoForm

À terme, un factory createAutoForm assemblera toutes les dépendances : 

// Utilisation finale — l'utilisateur ne voit que le schema
const Form = createAutoForm({
  adapter: tanstackAdapter,
  ui: shadcnAutoFormUi,
})

// Usage
<Form schema={myJsonSchema} />

AutoFormUi est le bloc UI interchangeable. L'utilisateur avancé peut en créer un custom ; l'utilisateur simple reçoit un default.

3 approches pour AutoFormUi

A — Slots nommés explicite

  • TextField, NumberField, BooleanField…
  • dispatch hardcodé dans auto-form-builder
  • Override d'un seul champ sans tout réécrire
  • Moins flexible (types figés)

B — Field unique flexible

  • 1 seul composant Field
  • reçoit fieldMeta pour dispatch interne
  • Flexibilité totale par thème
  • Override = tout réécrire

C — resolveField registry-like

  • resolveField(fieldMeta) → Component
  • Similaire à FieldRegistry aujourd'hui
  • Flexibilité maximale
  • Complexité pour l'utilisateur

Détail — Approche A : Slots nommés

interface AutoFormUi {
  // Structural
  Root: ComponentType<{
    children: ReactNode
    schema: Record<string, unknown>
    fields: FieldMeta[]
  }>
  ObjectField: ComponentType<ObjectFieldProps>
  ArrayField: ComponentType<ArrayFieldProps>
  CompositionsField: ComponentType<CompositionsFieldProps>

  // Leaf field — un slot par type
  TextField: ComponentType<FieldProps>
  NumberField: ComponentType<FieldProps>
  BooleanField: ComponentType<FieldProps>
  EnumField: ComponentType<FieldProps>
  TextareaField: ComponentType<FieldProps>
  SwitchField: ComponentType<FieldProps>

  // Submit
  SubmitButton: ComponentType<SubmitButtonProps>
}

Le dispatch est fait par auto-form-builder :

function resolveField(ui: AutoFormUi, meta: FieldMeta): FieldComponent {
  if (meta.kind === "enum")        return ui.EnumField
  if (meta.uiWidget === "textarea") return ui.TextareaField
  if (meta.uiWidget === "switch")   return ui.SwitchField
  if (meta.type === "string")       return ui.TextField
  if (meta.type === "number"
   || meta.type === "integer")      return ui.NumberField
  if (meta.type === "boolean")      return ui.BooleanField
  return ui.TextField
}
Critère A — Slots B — Field unique C — resolveField
Override partiel ✓ un slot à la fois ✗ tout réécrire ~ si helper fourni
Dispatch personnalisé ✗ fixé dans la lib ✓ dans le composant ✓ fonction publique
Charge cognitive ✓ faible ~ moyenne ✗ plus élevée
Nbre de propriétés 11 6 6
Extensibilité (types customs) ✗ limité ✓ totale ✓ totale
Testabilité unitaire ✓ triviale ✓ bonne ✓ bonne

Analyse

Pourquoi A (Slots nommés) semble le meilleur compromis

  1. Override partiel — l'utilisateur qui veut juste changer TextField en Textarea ne réécrit rien d'autre. Avec B ou C, il doit soit tout réécrire, soit implémenter un dispatch complet.
  2. Charge cognitive minimale — le nom du slot dit exactement ce qu'il fait. Pas besoin de comprendre le système de matching du FieldRegistry.
  3. Compatible avec createAutoForm — la factory prend un Partial<AutoFormUi> et complète avec les defaults. L'utilisateur passe juste { TextField: MyCustomText }.
  4. DX similaire à shadcn/ui — les composants shadcn s'utilisent individuellement. Même philosophie.

Le compromis : extensibilité

Si plus tard un utilisateur a besoin d'un type complètement nouveau (ex: color-picker, markdown-editor), les slots nommés seuls ne suffisent pas. Solution : ajouter un slot customFieldResolver?: (meta: FieldMeta) => FieldComponent | null optionnel, appelé avant le dispatch par type. Simple, extensible, non-cassant. 

// Version finale avec escape hatch optionnel
interface AutoFormUi {
  Root: ComponentType<RootProps>
  ObjectField: ComponentType<ObjectFieldProps>
  ArrayField: ComponentType<ArrayFieldProps>
  CompositionsField: ComponentType<CompositionsFieldProps>

  TextField: ComponentType<FieldProps>
  NumberField: ComponentType<FieldProps>
  BooleanField: ComponentType<FieldProps>
  EnumField: ComponentType<FieldProps>
  TextareaField: ComponentType<FieldProps>
  SwitchField: ComponentType<FieldProps>

  SubmitButton: ComponentType<SubmitButtonProps>

  // Escape hatch — optionnel, pour les types customs
  resolveField?: (meta: FieldMeta) => FieldComponent | undefined
}

Migration

Suppression de @code2-base-ui/ui

  1. layout/shadcn.tsx + fields/field-components.tsx + fields/index.ts sont extraits dans un package externe (ou gardés dans l'app)
  2. auto-form-builder ne dépend plus que de json-schema-toolkit
  3. Un shadcnAutoFormUi par défaut est fourni (dans un nouveau package ou dans l'app)
  4. auto-form.tsx utilise AutoFormUi au lieu de FormLayoutCtx + registry

Recommandation

Approche A (Slots nommés) + escape hatch resolveField optionnel.

Raisons : override partiel, charge cognitive minimale, compatible avec la vision createAutoForm, extensible sans breaking change.

FieldSet, FieldGroup, FieldLegend, FieldDescription disparaissent de l'interface publique — absorbés par Root (pour le wrapper haut niveau) et par ObjectField/ArrayField (pour les wrappers structurels).
